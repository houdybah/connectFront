Vue d'ensemble

DouaneConnect joue le rôle d'un fournisseur d'identité centralisé pour l'ensemble des applications de la Direction Générale des Douanes, un peu à la manière de Keycloak :

  • Application (ce que Keycloak appelle un client) : chaque système métier (ALTEREGO, COEA, DACTOV, SYDONIA WORLD, SYGDRD, SYGREH, E-DOUANE, ...) est déclaré une fois dans DouaneConnect avec un code unique, une url et un statut enabled.
  • Profil (équivalent d'un rôle/realm role) : défini par application, il regroupe un ensemble de Privilèges (permissions fines).
  • Utilisateur : un compte unique dans DouaneConnect, auquel on attache un ou plusieurs Profils (donc un accès à une ou plusieurs applications) via la relation UserProfile (hasAccess, enabled, nonLocked, nonExpired).
  • Authentification déléguée : les applications clientes ne gèrent pas leurs propres mots de passe. Elles redirigent l'utilisateur vers DouaneConnect (ou lui envoient email/mot de passe côté serveur), reçoivent un token, puis l'échangent contre une session/JWT local.

En résumé : 1 application = 1 client, 1 profil = 1 rôle, 1 privilège = 1 permission, exactement comme dans un realm Keycloak.

Sécurité des tokens (chiffrement de bout en bout)

Le token DouaneConnect ne circule jamais en clair : un JWT est généré puis immédiatement chiffré (AES-256-GCM) avant de quitter le serveur. C'est cette forme chiffrée qui est persistée en base (Utilisateur.currentToken), renvoyée par /authenticate, utilisée comme Authorization: Bearer ... pour tous les appels à l'API DouaneConnect, transmise en paramètre ?token=... à l'application cliente affichée dans l'iframe, et renvoyée à /api/token/validate/simple pour validation. Le JWT en clair n'existe que le temps d'un traitement serveur.

Deux couches de chiffrement
  1. Serveur → client (obligatoire) : TokenEncryptionUtil (AES-256-GCM, clé dérivée par SHA-256 de token.encryption.secret). Déchiffré côté serveur par JwtRequestFilter (pour authentifier les appels API) et par TokenValidationServiceImpl (pour la validation SSO).
  2. Stockage navigateur (défense en profondeur) : avant d'écrire le token (déjà chiffré côté serveur) dans sessionStorage, TokenStorageService lui applique une seconde couche AES (CryptoJS, clé environment.encryptionSecretKey), pour qu'il ne soit pas lisible tel quel via les outils de développement du navigateur.
Déchiffrement avant utilisation
  • TokenStorageService.getToken() ne fait que retirer la couche "stockage navigateur" : il renvoie le token toujours chiffré côté serveur — c'est la valeur correcte à renvoyer telle quelle à l'API DouaneConnect ou à transmettre à une application cliente.
  • Pour lire un champ du token (rôle, profils, jti...), il faut d'abord le déchiffrer : await TokenStorageService.getDecryptedClaims() effectue le déchiffrement AES-256-GCM (Web Crypto API, compatible avec TokenEncryptionUtil) puis décode le JWT obtenu.
  • Côté backend, tout endpoit qui reçoit un token (Bearer ou ?token=) doit appeler tokenEncryptionUtil.decrypt(...) avant toute lecture JJWT — déjà fait dans JwtRequestFilter et TokenValidationServiceImpl.

Important : token.encryption.secret (backend, application.properties) et environment.serverTokenSecret (frontend) doivent être strictement identiques. En production, ces deux valeurs doivent provenir d'une configuration sécurisée (variable d'environnement / coffre-fort de secrets), pas d'une valeur codée en dur.

Intégrer une nouvelle application
Étape 1 — Déclarer l'application dans DouaneConnect

Depuis Applications (ou par API), créez l'application cliente : 

POST /application
{{ '{' }}
  "code": "MON_APP",
  "nom": "Mon Application",
  "url": "https://mon-app.gov.gn",
  "environnement": "Production",
  "description": "Description fonctionnelle de l'application",
  "enabled": true
{{ '}' }}
Étape 2 — Définir les profils et privilèges

Toujours pour cette application, créez au moins un profil (ex. Administrateur, Utilisateur standard) puis les privilèges à lui rattacher, depuis la page Applications (onglet Profils / Privilèges) : 

POST /privilege   { "nom": "GESTION_DOSSIER", "description": "...", "application": { "uuid": "..." } }
POST /profile     { "nom": "Administrateur", "application": { "uuid": "..." } }
POST /profile/{uuidProfile}/privilege/{uuidPrivilege}   (attache un privilège au profil)
Étape 3 — Configurer le backend de l'application cliente

Dans application.properties de la nouvelle application :

api.external.base-url=https://douaneconnect.gov.gn/api

Puis exposer un endpoint qui valide le token reçu de DouaneConnect et ouvre une session locale :

@PostMapping("/authenticateWithDouaneConnect")
public ResponseEntity<?> authenticate(@RequestParam String token) {{ '{' }}
    ValidateTokenResponse result = apiConsumerService.validerTokenSimpleSync(token);
    if (Boolean.TRUE.equals(result.getValid())) {{ '{' }}
        String email = result.getUtilisateur().getEmail();
        UserDetails userDetails = userDetailsService.loadUserByUsername(email);
        String jwt = jwtTokenUtil.generateToken(userDetails);
        return ResponseEntity.ok(new JwtResponse(jwt));
    {{ '}' }}
    return ResponseEntity.status(HttpStatus.UNAUTHORIZED).build();
{{ '}' }}

Vérifiez systématiquement que l'utilisateur a bien accès à codeApp avant d'ouvrir une session (champ hasAccess du profil renvoyé dans userAppDtos).

Étape 4 — Brancher le frontend (application affichée dans un frame DouaneConnect)
  1. Depuis le portail (/modules/applications), un clic sur une carte application ouvre celle-ci dans un iframe intégré à DouaneConnect (route /modules/app-frame) plutôt que de rediriger la page entière.
  2. L'URL chargée dans l'iframe est {{ '{' }}urlApplication{{ '}' }}?token=...&codeApp=...&url=... : le token DouaneConnect, le codeApp de l'application cliquée, et une url de retour vers le portail.
  3. L'application cliente détecte ces paramètres dès le chargement, appelle son backend qui valide le token auprès de DouaneConnect (codeApp inclus, voir Étape 3), puis ouvre sa propre session applicative.
  4. Important : pour être affichable dans l'iframe, l'application cliente ne doit pas envoyer d'en-tête X-Frame-Options: DENY ni de Content-Security-Policy: frame-ancestors restrictive.
Étape 5 — Tester

Utilisez Swagger (/swagger-ui.html) côté DouaneConnect et côté application cliente pour valider le flux de bout en bout, avec un utilisateur de test possédant un profil sur la nouvelle application.

Administrer les utilisateurs et les accès
Accès à l'espace admin scopé par application : un SUPER_USER voit tout l'espace /modules/admin. Un utilisateur portant le profil Administrateur sur une ou plusieurs applications — sans être SUPER_USER — peut aussi y accéder (AdminGuard côté frontend), mais n'y voit que ses propres applications (ApplicationListComponent), les utilisateurs qui y sont rattachés (UtilisateurListComponent), et des statistiques de tableau de bord scopées à ces applications. Ce filtrage est appliqué côté serveur (ApplicationServiceImpl, UtilisateurServiceImpl, DashboardServiceImpl consultent SecurityContextService.getAdminApplicationUuids()), pas seulement dans l'interface.
  1. Créer un utilisateur depuis Utilisateurs : email, nom, prénom, service/fonction, rôle (SUPER_USER ou USER_APP).
  2. Attribuer un accès applicatif : associer à l'utilisateur un ou plusieurs profils (un profil par application souhaitée). Cela crée un enregistrement UserProfile avec hasAccess = true.
  3. Gérer le cycle de vie du compte sans le supprimer, via les indicateurs du profil utilisateur : enabled (compte actif), nonLocked (non verrouillé), nonExpired (non expiré), hasAccess (accès à cette application spécifiquement révocable indépendamment des autres).
  4. Ajuster les droits fins depuis la gestion des profils/privilèges de chaque application : ajouter/retirer un privilège d'un profil impacte instantanément tous les utilisateurs qui portent ce profil.
  5. Suivre l'activité via Journaux d'Audit et Historique des Connexions.
Référence API rapide
Ressource Endpoint Usage
Applications GET/POST /application, PUT|DELETE /application/{uuid} Déclarer / lister / modifier / supprimer une application cliente
Profils GET/POST /profile, GET /profile/application/{uuidApplication} Créer / lister les profils (rôles) d'une application
Privilèges GET/POST /privilege, GET /privilege/application/{uuidApplication} Créer / lister les privilèges (permissions) d'une application
Attribution profil ↔ privilège POST|DELETE /profile/{uuidProfile}/privilege/{uuidPrivilege} Attacher / retirer un privilège d'un profil
Utilisateurs GET/POST /utilisateur, PUT|DELETE /utilisateur/{uuid} Gestion des comptes utilisateurs
Accès applicatifs d'un utilisateur GET /api/utilisateur/{uuidUtilisateur}/applications/active Lister les applications auxquelles un utilisateur a effectivement accès
Authentification email/mot de passe POST /api/validateTheAuthentication?email=&password=&codeApp= Utilisé par une application cliente pour authentifier un utilisateur pour son code app
Validation de token POST /api/token/validate/simple?token=&codeApp= Valide un token DouaneConnect. Si codeApp est fourni, la réponse est scopée à cette application (codeApplication, privileges) et refuse l'accès (valid: false) si l'utilisateur n'y a pas de profil actif. Retourne aussi attributs ([{{ '{' }}"cle": "...", "valeur": "..."{{ '}' }}]).
